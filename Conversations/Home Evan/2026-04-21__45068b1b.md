## What we're building

Right now, Hermes recovers context by reading `tail -n 150` of SESSION_LOG — the last ~150 lines, regardless of relevance. If you worked on something 3 weeks ago and come back to it, that entry might be 800 lines back and completely invisible.

ChromaDB adds a semantic retrieval layer: instead of recency-based recall, Hermes can ask *"what do I know about NavMesh setup?"* or *"what did we decide about beat timing?"* and get the relevant past entries back regardless of when they were written.

## Design decisions to nail down first

**1. What gets indexed?**
- Just `SESSION_LOG.md` entries (most useful — these are the decisions/discoveries)
- All vault markdown files (broader but noisier)
- Both, in separate collections

**2. What triggers a query?**
- **Auto-prefetch** — before every turn, embed the user's message and pull relevant past entries. Always-on but adds latency and could be noisy for short exchanges.
- **Manual tool** — Hermes gets a `search_vault(query)` tool it can call when it thinks past context is relevant. More deliberate, less automatic.
- **Session-start only** — query once at the start of each session based on the current topic. Lightweight but you only get one shot.
- **Combination** — manual tool + session-start query (probably the right answer)

**3. What triggers indexing?**
- After every SESSION_LOG write (the obsidian-vault plugin writes → immediately index the new entry)
- On session start (diff SESSION_LOG against what's already in ChromaDB, index new entries)
- Both (session-start catch-up + real-time write hook)

**4. Embedding model?**
On an RPi5 (aarch64), options are:
- `sentence-transformers` with `all-MiniLM-L6-v2` — ~90MB, runs on CPU, ~200ms per chunk. Practical.
- ChromaDB's default (also uses sentence-transformers under the hood)
- Call the LLM's embedding endpoint if MiniMax exposes one — no extra model weight

**5. ChromaDB mode?**
- **Embedded** (just a local directory, no server) — simplest, only accessible from the same process
- **Server mode** — runs as a separate process, accessible over HTTP. Lets multiple agents or tools query it independently.
