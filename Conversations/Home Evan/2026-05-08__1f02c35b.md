### Layer 4 — RAG Query Layer


## Assistant

Think of it like building an index for a library — but smarter than alphabetical order.

When you embed a document, an AI model reads it and converts it into a list of numbers (a "vector") that captures the *meaning* of the text. Documents about similar topics end up with similar numbers, even if they use different words.

**Concrete example:** If you embed these three things:
- An email saying *"carrier is disputing O&P on Cordes claim"*
- A battle book section on *"arguments for overhead and profit"*
- An old email where you won an O&P argument against Allstate

...all three end up numerically close to each other in the database, because they're all about the same concept. So when you ask *"what's worked against Allstate on O&P?"*, the system finds all three — even though the words aren't identical.

**Why it matters for your use case:** Your 20k emails, PDFs, and battle book are sitting as raw files right now. Until they're embedded, I can't search them by meaning — I'd have to read every single file every time you ask something. After embedding, the whole corpus is indexed and I can find the 6 most relevant chunks out of 50,000+ in milliseconds.

The download step (happening now) just saves the attachment files to disk. The embed step (next) is what turns those files into a searchable knowledge base.

## User

so how does this work with 7 workers now?

## Assistant

Each worker gets assigned roughly 1/7th of the files, split by index. So if there are 20,215 emails:

- Shard 0 gets files 0, 7, 14, 21... 
- Shard 1 gets files 1, 8, 15, 22...
- Shard 2 gets files 2, 9, 16, 23...
- ...and so on

They all run completely independently — each one reads its own slice of emails, extracts text, runs it through the embedding model on its own CPU core, and writes to ChromaDB. They don't know about each other at all.

The only shared resource is ChromaDB, which handles the 7 workers writing to it simultaneously. Since each worker is writing different IDs (different emails), there's no collision — they just take turns on the write lock, which is fast on NVMe.

Same thing happens for the attachment second pass — each shard handles 1/7th of the attachment directories.

Net result: roughly 7x faster than a single worker, minus a small amount of overhead from the disk and ChromaDB contention.

## User

ok, what happens if there IS an overlap with a uuid

## Assistant

Nothing bad — ChromaDB uses `upsert`, not `insert`. If two workers happen to write the same ID, the second one just overwrites the first with identical data. No duplicates, no crash, no corruption. It's a no-op in practice.

The IDs are MD5 hashes of the Gmail message ID, so the same email always produces the same ID. Two workers could theoretically land on the same email if the shard math had a bug, but the data would still be correct — just written twice.

## User

what happens in the case that multiple email threads are concerning the same claim

## Assistant

They all get stored separately but tagged with the same `claim_id` in their metadata. So in ChromaDB you'd have say 12 separate chunks all with `claim_id: PROPX2022014806` — each one a different email or attachment from that claim.

When you query for that claim, ChromaDB filters by `claim_id` and returns all of them. The semantic search then ranks them by relevance to your specific question. So if you ask "what did USAA say about depreciation on the Cordes claim", it finds all 12 chunks, ranks the ones about depreciation highest, and I synthesize an answer from those.

The thread_id works similarly — all emails in the same Gmail conversation share a `thread_id`, so I can reconstruct the full back-and-forth of a specific exchange if needed. Claim_id captures everything across the whole claim (multiple threads, attachments, different time periods), thread_id captures a single conversation within it.
