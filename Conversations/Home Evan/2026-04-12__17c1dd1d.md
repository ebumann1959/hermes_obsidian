## Most Common Causes

**1. Firmware too old for 16GB variant**
The Pi 5 16GB requires a relatively recent kernel/firmware. If your OS image predates the 16GB release (~late 2024), it may not boot or may only recognize 8GB.

**2. Bootloader EEPROM mismatch**
Each Pi 5 unit has its own EEPROM with a bootloader. The new board's EEPROM version may conflict with how the SD was set up.

## How to diagnose

First, clarify what you mean by "does not boot as expected":
- **No output at all / solid green LED?** → likely EEPROM or power issue
- **Boots but only 8GB RAM?** → firmware too old
- **Boots but crashes/unstable?** → probably needs updates

## Fixes to try

**If it partially boots or you can SSH in:**
```bash
sudo apt update && sudo apt full-upgrade -y
sudo rpi-eeprom-update -a
sudo reboot
```

**If it won't boot at all:**
1. Use [Raspberry Pi Imager](https://www.raspberrypi.com/software/) on another machine to write a fresh Raspberry Pi OS (64-bit) to a spare card
2. Boot the new Pi 5 with that fresh card first — this ensures the EEPROM is updated
3. Then try your original SD card again

**Check EEPROM bootloader date on the new Pi:**
```bash
sudo rpi-eeprom-update
```
You want a date of **2024-09-23 or newer** for full 16GB support.

## Quick Fix — Edit cmdline.txt from Another Machine

The boot partition is FAT32, readable on any OS (Windows, Mac, Linux).

1. **Pull the SD card** and plug it into another computer
2. Open the **boot partition** (small FAT32 partition, may show as `bootfs` or similar)
3. Open `cmdline.txt` — it's a single long line, something like:
   ```
   console=serial0,115200 console=tty1 root=PARTUUID=... rootfstype=ext4 ... quiet
   ```
4. **Append to the end of that line** (don't add a new line):
   ```
   init=/bin/bash
   ```
5. Save, put SD back in the Pi, boot

This bypasses systemd and drops you directly into a root shell.

## Once in the Shell

Run:
```bash
mount -o remount,rw /
```
Then check what's actually failing:
```bash
journalctl -xb        # if journald is available
cat /etc/fstab        # look for any bad entries
```

The most common cause of this on a new board is a **bad fstab entry** or a service that can't find hardware it expects. Share what you see in fstab or the journal and I can help pinpoint it.

## After Fixing

Remove `init=/bin/bash` from `cmdline.txt` and reboot normally.

## Assistant

Found it. Your fstab has this entry:

```
UUID=06df3512-57d0-44a0-9e0c-52be300fb619 /mnt/ssd ext4 defaults,noatime 0 2
```

That external SSD isn't connected to the new Pi, so the mount fails at boot → emergency mode → locked root → stuck.

**Fix:** Add `nofail` so it skips the mount gracefully if the drive isn't present. Run this in your terminal:

```bash
sudo sed -i 's|defaults,noatime 0 2|defaults,noatime,nofail 0 2|' /mnt/piroot/etc/fstab
```

Then verify it looks right:
```bash
cat /mnt/piroot/etc/fstab
```

The last line should now read `...defaults,noatime,nofail 0 2`. After that, unmount and you're done:

```bash
sudo umount /mnt/piboot /mnt/piroot
```

Put the SD back in the Pi 5 — it should boot normally. When you reconnect the SSD it'll mount automatically; if it's not there, it'll just skip it.
